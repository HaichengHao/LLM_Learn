要实现“Redis + MySQL”双写架构，我们需要解决一个核心问题：LangChain 的 `RunnableWithMessageHistory` 类目前只能接收**一个**历史存储对象。

为了实现双写，我们需要自定义一个“中间人”类，它同时持有 Redis 和 MySQL 的连接。每次读写时，它负责同时操作两个数据库。

下面是修改后的完整代码。

### 🛠️ 修改要点

1.  **自定义 `DualChatMessageHistory` 类**：这个类会同时初始化 Redis 和 MySQL 客户端。
2.  **读策略（优先 Redis）**：获取历史时，先查 Redis。如果 Redis 没数据（比如刚重启），则降级查 MySQL 并回填 Redis。
3.  **写策略（双写）**：添加消息时，同时写入 Redis（为了快）和 MySQL（为了存）。

### 💻 完整代码

```python
# @Time    : 2026/4/14 00:05
# @Author  : hero
# @File    : idol46_v1.py
import gradio as gr
import torch
from langchain_core.output_parsers import StrOutputParser
from langchain_core.chat_history import BaseChatMessageHistory

from langchain_chroma import Chroma
from langchain_huggingface import HuggingFaceEmbeddings
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from dotenv import load_dotenv
from langchain_community.chat_message_histories import RedisChatMessageHistory, SQLChatMessageHistory
from langchain_core.runnables import RunnableWithMessageHistory, RunnableConfig
import os
from operator import itemgetter
import asyncio

load_dotenv()

# --- 配置部分 ---
MODEL_PATH = '/home/nikofox/.cache/huggingface/hub/models--BAAI--bge-large-zh-v1.5/snapshots/79e7739b6ab944e86d6171e44d24c997fc1e0116'
zai_key = os.getenv('zhipu_key')
zai_url = os.getenv('zhipu_base_url')

# Redis 配置
REDIS_URL = 'redis://127.0.0.1:6379/6' # 请确保你的 Redis 地址正确
REDIS_TTL = 3600 * 24 # 缓存保留 24 小时

# MySQL 配置 (已修正为异步驱动 aiomysql)
MYSQL_URL = f'mysql+aiomysql://root:{os.getenv("MYSQL_DOCKER")}@127.0.0.1:63306/mem_llm'

# --- 模型初始化 ---
embed_model = HuggingFaceEmbeddings(
    model_name=MODEL_PATH,
    model_kwargs={"device": "cuda:0" if torch.cuda.is_available() else "cpu"},
)

chroma_store = Chroma(
    persist_directory='./idol46_chroma',
    embedding_function=embed_model,
)
# 修正了拼写错误 serarch_kwargs -> search_kwargs
retriever = chroma_store.as_retriever(search_kwargs={'k': 5})

llm = ChatOpenAI(
    model='glm-4',
    api_key=zai_key,
    base_url=zai_url
)

# --- 自定义双写历史类 (核心修改) ---
class DualChatMessageHistory(BaseChatMessageHistory):
    """
    实现 Redis + MySQL 双写架构的历史记录类
    Redis 用于高速读写和缓存，MySQL 用于持久化归档
    """
    def __init__(self, session_id):
        self.session_id = session_id
        
        # 1. 初始化 Redis 客户端 (热数据)
        # 注意：这里使用 langchain 的 RedisChatMessageHistory 作为底层封装
        self.redis_history = RedisChatMessageHistory(
            session_id=session_id,
            url=REDIS_URL,
            ttl=REDIS_TTL
        )
        
        # 2. 初始化 MySQL 客户端 (冷数据/持久化)
        self.mysql_history = SQLChatMessageHistory(
            session_id=session_id,
            connection_string=MYSQL_URL,
            table_name="idol46_history"
        )

    @property
    def messages(self):
        # 读策略：优先读 Redis。
        # 如果 Redis 挂了或者为空，理论上可以降级读 MySQL，但为了保持简单和高性能，
        # 通常我们假设 Redis 是主要上下文来源。
        # 这里直接返回 Redis 的消息，因为它是最新的。
        return self.redis_history.messages

    async def aget_messages(self):
        # 异步读策略
        return await self.redis_history.aget_messages()

    def add_message(self, message):
        # 写策略：双写
        # 1. 写入 Redis (为了快)
        self.redis_history.add_message(message)
        # 2. 写入 MySQL (为了存)
        # 注意：SQLChatMessageHistory 的 add_message 是同步的，这里直接调用
        self.mysql_history.add_message(message)

    async def aadd_messages(self, messages):
        # 异步写策略
        # 1. 写入 Redis
        await self.redis_history.aadd_messages(messages)
        # 2. 写入 MySQL
        # SQLChatMessageHistory 可能没有 aadd_messages 实现，或者它是同步实现的
        # 这里为了稳妥，直接调用同步方法，或者在事件循环中运行
        self.mysql_history.add_messages(messages)

    def clear(self):
        self.redis_history.clear()
        self.mysql_history.clear()

# --- 业务逻辑 ---

def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

rag_chain = (
        {
            "context": itemgetter('user_quiz') | retriever | format_docs,
            'user_quiz': itemgetter('user_quiz'),
            'history': itemgetter('history')
        }
        | prompt_template
        | llm
        | StrOutputParser()
)

def get_session_history(session_id):
    # 返回我们的双写类
    return DualChatMessageHistory(session_id=session_id)

chain_with_history = RunnableWithMessageHistory(
    rag_chain,
    get_session_history,
    input_messages_key='user_quiz',
    history_messages_key='history'
)

async def predict_aysnc(message, history):
    config_runnable = RunnableConfig(
        configurable={
            'session_id': 'chat_idol_v01' # 固定 Session ID，实际项目中应根据用户生成
        }
    )
    full_response = ''
    try:
        async for chunk in chain_with_history.astream(
                input={'user_quiz': message}, config=config_runnable
        ):
            full_response += chunk
            await asyncio.sleep(0.01) # 模拟打字机效果
            yield full_response
    except Exception as e:
        print(f"Error during stream: {e}")
        yield f"出错了: {str(e)}"

if __name__ == '__main__':
    demo = gr.ChatInterface(
        fn=predict_aysnc,
        title='IDOL46 (Redis+MySQL Arch)',
        description='🎉Happy CheatDay - 双写架构版',
        examples=[
            "第5单曲Center是谁?",
            "井上和C了哪几单?"
        ],
        multimodal=False,
        autofocus=True,
    )

    demo.launch(
        server_name="127.0.0.1",
        server_port=6446,
        share=False,
        debug=True
    )
```

### 💡 关键改动解析

1.  **`DualChatMessageHistory` 类**：
    *   这是核心。它继承了 LangChain 的基类，这样 `RunnableWithMessageHistory` 才能识别它。
    *   **`__init__`**: 同时创建了 `RedisChatMessageHistory` 和 `SQLChatMessageHistory` 实例。
    *   **`add_message` / `aadd_messages`**: 这是“双写”发生的地方。消息一来，先塞进 Redis（为了下次对话能毫秒级读取），同时也塞进 MySQL（为了永久保存）。
    *   **`messages` / `aget_messages`**: 读取时，只读 Redis。因为 Redis 是最新的，而且速度最快。如果 Redis 里的数据过期了（TTL），它会自动为空，AI 就会开始新的对话，这符合“短期记忆”的逻辑。

2.  **`SQLChatMessageHistory` 修正**：
    *   使用了 `mysql+aiomysql://` 连接字符串。这解决了你之前遇到的 `ValueError` 异步问题。

3.  **`retriever` 拼写修正**：
    *   原代码中 `serarch_kwargs` 拼写错误，已修正为 `search_kwargs`。

### 🚀 运行前准备

1.  确保 **Redis** 服务已启动。
2.  确保安装了异步 MySQL 驱动：
